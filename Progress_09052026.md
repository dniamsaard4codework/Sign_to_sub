## 1. แรงจูงใจของ Ablation นี้

`Gloss` → `Gloss_Input` เพราะ `Gloss_Input` เป็น tier ที่ทีมงาน curate

## 6. Ground-truth leakage check (สำคัญ!)

ก่อนเชื่อผล "Gloss ชนะทุกอย่าง" ต้องตรวจว่า gap ใหญ่ของ exact-text-match
**ไม่ใช่ data leakage**

### 6.1 ตรวจ token sequence vs GT sequence

flatten tokens จากแต่ละ tier แล้วเทียบ position-by-position กับ
`Gloss Labeling` (852 entries) — สมมติว่า annotator สร้าง GT จาก tier
ใด tier หนึ่ง ทั้งสองควรมี token sequence ใกล้เคียงกัน

| Comparison | #ตรงกัน positional / ทั้งหมด | % match |
| --- | --- | --- |
| `Gloss` tokens vs GT texts | 607 / 852 | **71.2%** |
| `Gloss_Input` tokens vs GT texts | 42 / 852 | 4.9% |

**ตีความ:**
1. `Gloss` token-list มี **71.2% positional identity** กับ GT —
   หมายความว่า annotator น่าจะใช้ `Gloss` เป็นจุดตั้งต้นในการ build
   GT (split sentences ตาม token เดิม + แก้ token text บางส่วน)
2. `Gloss_Input` มี **เพียง 4.9%** — เพราะมี 37 ขอบ token ที่ต่างกัน
   (split/merge ใหม่) ทำให้ทุก position หลังจุดต่างแรกเลื่อนตำแหน่ง
3. ดังนั้น **65.10% exact-text-match ของ Gloss ส่วนหนึ่งเกิดจาก
   structural alignment ของ tokenization** ไม่ใช่ความสามารถของโมเดล
   ในการ "เลือก token ที่ถูกที่"

### 6.2 ผลกระทบต่อการตีความ metric อื่น

| Metric | กระทบจาก leakage? |
| --- | --- |
| Mean / Median IoU | **กระทบเล็กน้อยทางอ้อม** — token boundary ของ Gloss ใกล้ GT มากกว่า ทำให้ DP จับช่วงเวลาที่ใกล้ "ขนาดจริง" ได้ง่ายขึ้น |
| % IoU ≥ 0.5 | **กระทบทางอ้อม** เหมือน mean IoU |
| % any overlap | กระทบน้อย — ทั้ง 2 รันมี sentence-level window pad เหมือนกัน |
| Exact text match | **กระทบหนักที่สุด** — token text เป็นการเปรียบเทียบสตริงตรงๆ |

### 6.3 Metric ที่ "fair" ที่สุดในการเทียบ

- **% IoU ≥ 0.3 / 0.5** บนการเทียบ best-IoU pairing — ไม่อิง token text
- **Mean absolute start/end offset (sec)** — วัด timing precision

ทั้งสอง metric **Gloss ก็ยังชนะ** (`+10.97 pp` และ offset ตัวเลขดีกว่า
ใน §7) ดังนั้นแม้ระวัง leakage แล้ว **ผล temporal accuracy ของ
Gloss ยังดีกว่า** อย่างมีนัยสำคัญ

## 9. มุมมองแบบ "per-GT" (coverage)

โหมดเดิมของ `evaluate_gloss_labeling.py` วน "ทุก prediction" — แต่ถ้า
มี predictions มากกว่า GT (ในกรณี Gloss_Input คือ 889 vs 852) จะมี
ปัญหาว่า predictions ส่วนเกินอาจจะถูกผูกกับ GT entry ผิดอันได้

มุมมอง **per-GT** ตอบคำถามต่างกัน: **"GT แต่ละอันได้รับการ cover
จาก prediction ที่ดีที่สุดเท่าไหร่"**

| Metric | **Gloss** (852 GT) | **Gloss_Input** (852 GT) |
| --- | ---: | ---: |
| Mean best-IoU per GT | **0.4906** | 0.4230 |

- `Gloss_Input` ทิ้ง GT entries **11.03%** ที่ไม่มี prediction overlap

## 10. DP score distribution

`align_gloss_labels.py` คำนวณ score ต่อ token = sum of softmax-similarity
ในช่วง segments ที่จับให้ — สูง = โมเดลมั่นใจ

| Stat | Gloss | Gloss_Input |
| --- | ---: | ---: |
| Mean score | 0.1373 | 0.1292 |
| Median score | 0.1162 | 0.1048 |
| Min | 0.0194 | 0.0179 |
| Max | **1.0000** | 0.6411 |

**ตีความ:**
- Mean/median **Gloss สูงกว่าเล็กน้อย** — modest signal
- Max ของ Gloss แตะ 1.0 (มี case ที่ token ผูกอยู่กับ 1 segment เดียว
  ที่ similarity ครอบงำทั้ง row หลัง softmax) ในขณะที่ Gloss_Input ไม่ถึง
- ความสอดคล้องระหว่าง score กับ IoU ยังต่ำในทั้งสองกรณี — บอกว่า
  similarity score ของ SignCLIP ยังไม่ใช่ตัวพยากรณ์ alignment quality
  ที่ดี (เคยเห็นใน Progress_04052026 เช่นกัน)

## 11. Discussion: ทำไม Gloss ชนะ?

### 11.1 สมมติฐานที่ใช่ (high confidence)

1. **Token boundary ของ Gloss ใกล้ GT มากกว่า** — 71.2% positional
   token-identity บอกว่า annotator ใช้ Gloss เป็น base ในการสร้าง GT
   ดังนั้น token ของ Gloss แต่ละตัวจะมี "pre-image" อยู่แล้วใน GT
2. **Token count ตรงกับ GT (852=852)** — ทำให้ DP มี degree of freedom
   ตรงกับขนาดที่ "ถูกออกแบบมา" — `Gloss_Input` มี 889 tokens บีบใส่
   852 GT entries ทำให้บางส่วนต้องแชร์/ทับ

### 11.2 สมมติฐานที่ใช่บางส่วน (medium confidence)

3. **Sentence timestamps ของ Gloss ครอบ GT entries ได้ดีกว่า** — total
   duration: Gloss 560.55s vs Gloss_Input 541.25s; ใหญ่กว่า ~19s ช่วย
   ให้ window cover GT ที่ใกล้ขอบเขตของประโยคได้มากกว่า
   - หลักฐานเสริม: %any-overlap (per-GT) = 97.77% vs 88.97%
   - 8.97 pp ของ GT entries ของ Gloss_Input ตกอยู่ **นอก window**
     ของประโยคใดประโยคหนึ่งเลย แม้หลัง 0.5s padding

### 11.3 สมมติฐานที่ยังไม่สรุป

4. ไม่ชัดว่า Gloss_Input มี curated text "ดีกว่า" จริงหรือไม่ในเชิง
   linguistics — ในด้าน alignment กับ SIGN segments มันให้ผลแย่กว่า
   Gloss แต่ในด้าน CC alignment (Task 1) อาจให้ผลต่างออกไป — Task 1
   ไม่ได้ใช้ Gloss tier โดยตรง (ใช้ CC_Input) จึงไม่กระทบ

## 12. Recommendation

### 12.1 สำหรับ Task 2

**ถ้าใช้กับ Test.eaf นี้เพื่อ produce final aligned EAF:**
- **ใช้ `--tier Gloss`** เป็น default — ผลดีกว่า ทุก metric
- ปรับ default ของ `align_gloss_labels.py` กลับเป็น `Gloss` (ตอนนี้ยังเป็น
  `Gloss_Input` เพื่อความ backward compatibility กับ Progress_04052026)

**ถ้าจะใช้ Gloss_Input เป็น default จริง ต้อง:**
- review tokenization ของ Gloss_Input ให้สอดคล้องกับ Gloss Labeling
  (ลด 37 token ส่วนเกิน)
- ขยาย sentence window ของ tier ก่อน align (แทนที่จะใช้ window-pad =
  0.5s ของ aligner) เพราะปัจจุบัน Gloss_Input ทิ้ง GT 11% ที่ตกขอบ

### 12.2 สำหรับ benchmark ในอนาคต

- ผล **65.10% text match** ของ `Gloss` มี **leakage component** —
  อย่ารายงานเป็น "model accuracy" โดยไม่ disclose ที่มา
- รายงาน metric หลักเป็น **% IoU ≥ 0.3** หรือ **mean abs start
  offset** — สองตัวนี้ **immune ต่อ tokenization leakage**
- ในการเขียน paper / report ภายนอก: ต้องอธิบายว่า "Gloss tokens were
  used as the substrate for GT construction"

## 15. TL;DR

**Task 2 ablation: `Gloss` ชนะ `Gloss_Input` ทุก metric**

- Mean IoU: **0.4901 vs 0.4199** (+7 pp)
- % IoU ≥ 0.5: **48.4% vs 38.9%** (+9.4 pp)
- % IoU ≥ 0.3: **77.0% vs 66.0%** (+11.0 pp)
- % zero overlap: **2.5% vs 6.6%** (Gloss ครอบคลุม GT ดีกว่า)
- Mean abs start offset: **0.188s vs 0.212s** (Gloss แม่นกว่า)
- Exact text match: **65.1% vs 10.6%** (มี leakage component — ใช้
  metric IoU เป็นหลักดีกว่า)

**สาเหตุหลัก:** Gloss tokenization (852 tokens) สร้าง 1:1 mapping กับ GT
(852 entries) — ขณะที่ Gloss_Input มี 889 tokens (+37) ที่ผ่านการ
re-tokenize ทำให้ DP ต้อง align ใส่ขนาดที่ผิดธรรมชาติ

**Recommendation:** ใช้ `--tier Gloss` เป็น default ของ Task 2 — และเขียน
disclosed ว่า text-match metric มี structural advantage จากการที่
annotator ใช้ Gloss เป็น base ในการ build GT
