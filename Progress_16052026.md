## 1. แรงจูงใจของ Experiment นี้

ใน Progress_09052026 (ablation `Gloss` vs `Gloss_Input`) เราเห็นว่าการ
จับ token boundary ระดับ SIGN segment เป็น bottleneck ของ Task 2 —
sentence-level window ที่ใช้ filter SIGN candidates ครอบ GT ไม่เต็ม
(เช่น `Gloss_Input` ตก GT 11.03% นอกขอบ sentence window แม้หลัง 0.5s pad)

**สมมติฐาน:** ถ้าเรา crop วิดีโอเป็นต่อ sentence แล้ว run pose +
SEA segmentation ใหม่ใน scope ของประโยคเดียว — SIGN segment boundaries
ที่ได้จะเป็น "sentence-local" มากขึ้น (ไม่มี cross-sentence noise,
sign onset ที่เริ่มท้ายประโยคก่อนหน้าจะไม่ถูกจับ) — และผล Task 2
น่าจะดีขึ้น

## 2. Workflow ใหม่

```
04.mp4  ──crop (libx264, frame-accurate)──►  119 × clip_NNN.mp4
                                                    │
                                                    ▼
                                  videos_to_poses  →  119 × clip_NNN.pose
                                                    │
                                                    ▼
                          SEA/segmentation.py (E4s-1, batch over 119 ids)
                                                    │
                                                    ▼
                       extract_episode_features.py --mode=segmentation
                       (multilingual SignCLIP, batch over 119 ids)
                                                    │
                                                    ▼
       per-sentence DP (in-process; reuse align_one_sentence helper)
       sentence window = [0, clip_dur]; segments pool = SIGN ของ clip นั้น
                                                    │
                shift predicted times by +clip_start (recover original time)
                                                    │
                                                    ▼
              combined CSV / VTT / EAF + boundaries.csv (audit log)
                                                    │
                                                    ▼
                   evaluate_gloss_labeling.py vs Gloss Labeling
```

> **Key implementation choice:** batch ทุก clip ใน phase 3-5 (pose +
> seg + emb) แทนที่จะรัน per-clip subprocess 119 ครั้ง — ช่วยให้
> SignCLIP model load แค่ครั้งเดียว (ไม่ใช่ 119 ครั้ง) ทำให้ runtime
> phase 4-5 รวมกันแค่ ~7 นาที (ดู §7)

## 10. Discussion — ทำไมไม่ดีขึ้น?

### 10.1 สมมติฐานที่ตกไป

1. **"Sentence-local SIGN boundary จะแม่นกว่า"** — ผลจริง: ไม่แม่นกว่า
   ในภาพรวม — % zero overlap เพิ่ม +1.41 pp และ Mean abs start
   เพิ่ม 11 ms
2. **"Cross-sentence noise ใน whole-video segmentation จะทำให้ candidate
   pool สับสน"** — ผลจริง: window filter ของ aligner (`mid in [s, e]`)
   ใน baseline ก็ filter cross-sentence ออกเกือบหมดแล้ว — เพิ่ม layer
   ของ crop เลยไม่ได้กำไรเพิ่ม

### 10.2 สมมติฐานที่ใช่บางส่วน

3. **"ประโยคยาวจะได้กำไรจากการ crop"** — ผลจริง: ใช่ — bucket 13+ tokens
   ชนะ baseline +1.46 pp — ในประโยคยาว pose context จากนอกประโยคไม่ช่วย
   มากนัก แต่กลับเป็น noise
4. **"ประโยคสั้นจะเสียจากการ crop"** — ผลจริง: ใช่ และเสียหนัก
   −15.93 pp ใน bucket 1–3 tokens — clip < 3 s ขาด temporal context
   มากเกินไป

### 10.3 ที่มาของ regression เล็กน้อย

- **Frame-accurate libx264 re-encode** อาจ shift frame timestamps ในระดับ
  sub-frame (FPS=25 → 40 ms/frame) ทำให้ pose-derived sign boundaries
  เลื่อน ±1 frame
- **MediaPipe holistic** ต้องการ context หลายเฟรมเพื่อ smooth
  landmarks — clip สั้นทำให้ smoothing ทำงานไม่เต็มที่
- **SEA segmentation** GRU-based — hidden state เริ่มที่ zero ทุก clip
  ทำให้ sign แรกของแต่ละ clip ถูก detect ช้ากว่าใน whole-video pipeline
  ที่ hidden state warm จาก frame ก่อนหน้า

## 11. Recommendation

### 11.1 สำหรับ Task 2 production บน video 04

**อย่าใช้ per-sentence pipeline** — `--tier Gloss` แบบ whole-video เดิม
ยังเป็น best:
- Mean IoU 0.4901 (vs per-sentence 0.4763)
- Runtime ~10× น้อยกว่า
- Implementation เรียบง่ายกว่า (script เดียว ไม่ต้อง orchestrator)

### 11.2 สถานการณ์ที่ per-sentence อาจเหมาะ

- ถ้า input คือ **ประโยคยาว** (≥ 13 tokens) อย่างเดียว → per-sentence
  ให้ผลดีกว่าเล็กน้อย — แต่ subset แคบเกินไปสำหรับใช้จริง
- ถ้าทำกับ video set ที่ **มี sentence boundary ที่กำหนดมาจาก
  source ภายนอก** (ไม่ใช่ Gloss tier ที่มี timestamps อยู่แล้ว) — 
  per-sentence crop อาจช่วย sanity-check boundary

### 11.3 Future work

| ประเด็น | Action |
| --- | --- |
| Sub-3s clip regression | ลอง pad clip ก่อน-หลัง 1 s ก่อน pose extraction (overlap) แล้วครอป output back |
| MediaPipe context loss | ใช้ pose ที่ extract จาก whole video แทน แล้ว slice เฉพาะช่วง clip |
| SEA GRU hidden state cold start | warm-up segmentation ด้วย 0.5s ก่อนเริ่ม clip |
| ทำต่อ video อื่น | ต้อง refactor scripts หลายตัวให้รับ `--video_id` (ดู README §"Adapting for Multiple Videos") |

## 13. TL;DR

**Per-sentence video cropping pipeline แพ้ whole-video `Gloss` baseline เล็กน้อย:**

- Mean IoU: **0.4763 vs 0.4901** (−1.38 pp)
- % IoU ≥ 0.5: **46.0% vs 48.4%** (−2.35 pp)
- % zero overlap: **3.87% vs 2.46%** (worse +1.41 pp)
- Mean abs start offset: **0.199s vs 0.188s** (worse +11 ms)
- Exact text match: **60.0% vs 65.1%** (−5.10 pp)
- Runtime: **~69 min vs ~10 min** (~7× ช้ากว่า)

**แต่ยังชนะ whole-video `Gloss_Input` baseline ทุก metric** (Mean IoU
0.4763 vs 0.4199 → +5.64 pp)

**สาเหตุหลักที่ regress:**
1. ประโยคสั้น (1–3 tokens, clip < 3 s) เสียหนัก −15.93 pp — MediaPipe
   pose / SEA GRU ขาด temporal context
