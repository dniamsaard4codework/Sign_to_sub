## 1. บทนำและเป้าหมายโครงการ

โครงการนี้นำระบบ **SEA (Segment, Embed, and Align)** มาประยุกต์ใช้กับ **ภาษามือไทย (Thai Sign Language — TSL)** โดยเป้าหมายหลักคือ:

- **จัดตำแหน่งคำบรรยาย (subtitle) ให้ตรงกับช่วงเวลาที่ผู้แปลภาษามือแสดงท่าทาง** ในวิดีโอจริง
- ทดสอบว่า SEA ซึ่งพัฒนาสำหรับ BSL (British Sign Language) สามารถ **ทำงานข้ามภาษา (cross-lingual)** มาใช้กับ TSL ได้หรือไม่
- เปรียบเทียบประสิทธิภาพของ **โมเดล embedding หลายตัว** (BSL, Multilingual, ASL) กับข้อมูลภาษามือไทย
- สร้าง **เครื่องมือ pipeline ครบวงจร** ที่สามารถนำไปใช้ซ้ำกับวิดีโอภาษามือไทยตัวอื่นได้

---

### 🎯 เป้าหมายหลัก 2 งาน (Main Tasks)

โครงการนี้มี **2 งานหลัก** ที่ต้องแก้ไข:

#### งานที่ 1: CC (timestamp) → CC_Aligned (timestamp)

> **เป้าหมาย:** เลื่อนเวลาของคำบรรยาย (CC) ซึ่งตั้งตามจังหวะเสียงพูด ให้ไปตรงกับจังหวะที่ผู้แปลภาษามือแสดงท่าทาง (CC_Aligned)

**ปัญหาที่ต้องแก้:**
- คำบรรยาย CC มี timestamp จาก **เสียงพูดของผู้บรรยาย** (audio sync)
- แต่ผู้แปลภาษามือ **แสดงท่ามือช้ากว่าเสียงพูด** เนื่องจากต้องรับฟังก่อนแล้วจึงแปล
- CC มี 172 cues แต่ CC_Aligned (ground truth ที่นักวิจัย annotate ด้วยมือ) มี 119 entries เพราะบาง cue ไม่มีท่ามือสัมพันธ์กัน

**วิธีที่ SEA แก้ปัญหา:**
1. ตรวจจับช่วงเวลาที่มีท่ามือจริง (Segmentation) จาก pose data
2. ใช้ Dynamic Programming จับคู่ subtitle กับ sign segment ที่เหมาะสมที่สุด
3. ใช้ SignCLIP embedding วัดความคล้ายคลึงระหว่าง "ความหมายของท่ามือ" กับ "ข้อความใน subtitle"
4. อัปเดต timestamp ของ subtitle ให้ตรงกับ sign segment ที่ถูก assign

**ความเป็นไปได้ (Feasibility) — สูง ✅:**
- นี่คือ **งานหลักที่ SEA ออกแบบมาเพื่อทำ** โดยตรง — paper ต้นฉบับแสดงผลลัพธ์ที่ดีบน BSL, DGS, และ SGS
- ผลทดสอบกับ TSL ยืนยันว่าทำได้จริง: **80% ของ cues อยู่ภายใน ±1 วินาที** จาก ground truth (C_MULTI experiment)
- Sign language alignment เป็น **temporal reordering problem** ที่ DP แก้ได้อย่างมีประสิทธิภาพ
- Pose estimation ด้วย MediaPipe ไม่ขึ้นกับภาษามือ (language-agnostic) จึงทำงานกับ TSL ได้เลย
- SignCLIP (Multilingual variant) ถูก train บนหลายภาษามือ ทำให้ generalize มา TSL ได้แม้ไม่มี TSL training data

#### งานที่ 2: Gloss → Gloss Labeling

> **เป้าหมาย:** จัดตำแหน่ง gloss ระดับประโยค ("สวัสดี ผายมือ เด็ก เรียน") ลงไปเป็น annotation รายท่ามือแต่ละท่า (Gloss Labeling) โดยอัตโนมัติ

**ปัญหาที่ต้องแก้:**
- ปัจจุบัน **Gloss** (119 entries) มี timestamp ระดับประโยค — ระบุว่าช่วงไหนมีท่ามือชุดไหน
- **Gloss Labeling** (852 entries) คือ annotation ระดับท่ามือแต่ละท่า — ระบุว่า "ท่ามือ X เกิดที่วินาที Y ถึง Z"
- การทำ Gloss Labeling ด้วยมือเป็น **งานที่ใช้เวลามากที่สุด** ในการสร้าง sign language dataset
- ต้องการระบบอัตโนมัติที่แยกแต่ละคำใน Gloss text ออกมาแล้ว assign ให้กับ sign segment ที่ถูกต้อง

**วิธีที่ SEA สามารถนำมาประยุกต์ใช้:**
1. **Segmentation → ตรวจจับ sign segments (2,780 ท่ามือ)** จาก pose data — ส่วนนี้ทำได้แล้ว
2. **Tokenize Gloss text** — แยกแต่ละคำใน Gloss (เช่น "สวัสดี ผายมือ เด็ก เรียน" → 4 tokens)
3. **Embed แต่ละ token** ด้วย SignCLIP → ได้ embedding ของแต่ละคำ gloss
4. **คำนวณ similarity** ระหว่าง gloss token embedding กับ sign segment embedding
5. **DP alignment ระดับคำ** — assign แต่ละ gloss token ให้ sign segment ที่ similarity สูงสุดภายในช่วงเวลาของประโยคนั้น

**ความเป็นไปได้ (Feasibility) — ปานกลาง-สูง ⚠️✅:**

| ปัจจัย | สถานะ | หมายเหตุ |
| --- | --- | --- |
| Pose segmentation | ✅ ทำได้แล้ว | 2,780 sign segments — ครอบคลุมทุกท่ามือ |
| Sign embedding | ✅ ทำได้แล้ว | SignCLIP embed ทุก segment เป็น 768-dim vector |
| Gloss token embedding | ⚠️ ต้องพัฒนาเพิ่ม | SignCLIP text encoder รองรับ ภาษาอังกฤษ/สากล แต่ Gloss ภาษามือไทยใช้ **คำภาษาไทย** → ต้องทดสอบว่า Multilingual encoder รองรับหรือไม่ |
| Word-level DP alignment | ⚠️ ต้องปรับ pipeline | ปัจจุบัน align.py ทำงานระดับ "cue ↔ sign group" ต้องปรับเป็น "token ↔ sign segment" |
| Ground truth (Gloss Labeling) | ✅ มีอยู่แล้ว | 852 entries — ใช้เป็น ground truth สำหรับ evaluate |

**เหตุผลที่เป็นไปได้:**
- SEA pipeline มี **โครงสร้างพื้นฐานทุกอย่างแล้ว** (pose → segment → embed → align)
- DP alignment สามารถ **ลดขนาดปัญหาได้** โดยจำกัดขอบเขตการค้นหาเฉพาะ sign segments ภายในช่วงเวลาของ Gloss ประโยคนั้น (ไม่ต้องค้น 2,780 segments ทั้งหมด)
- C_MULTI experiment ใช้ Gloss text เป็น embedding input แล้วให้ผลดี → **SignCLIP เข้าใจ Gloss text ได้ในระดับหนึ่ง**
- Word-level similarity (`--tokenize_text_embedding`) ถูก implement ไว้แล้วใน `align_similarity.py` — pooling ด้วย mean/max
- Gloss Labeling มี 852 entries / 119 Gloss entries ≈ **7.2 ท่ามือต่อประโยค** ซึ่งเป็น search space ที่เล็กพอสำหรับ DP

**ความท้าทายที่ต้องแก้:**
- SignCLIP text encoder อาจจะไม่เข้าใจ gloss แต่ละคำในภาษาไทยดีเท่าระดับประโยค → อาจต้อง **fine-tune** หรือใช้ **gloss-to-English mapping**
- Segmentation อาจ **over-segment** (2,780 segments >> 852 labels) → ต้อง merge sign segments ที่อยู่ติดกันก่อน align
- DP parameters ต้อง **ปรับใหม่** สำหรับปัญหาระดับคำ (window size เล็กลง, gap penalty สูงขึ้น)

### ที่มาของ SEA

SEA เป็นงานวิจัยจาก University of Oxford และ ETH Zürich ตีพิมพ์ในปี 2025 ประกอบด้วย 3 ขั้นตอนหลัก:

1. **Segment** — ตรวจจับตำแหน่งท่ามือแต่ละท่าจาก pose data
2. **Embed** — แปลงท่ามือและคำบรรยายเป็น vector embedding ด้วย SignCLIP
3. **Align** — จับคู่คำบรรยายกับท่ามือด้วย Dynamic Programming (DP)

### 🔧 สรุปจุดที่ปรับได้ทั้ง Pipeline (Tunable Points Map — สำหรับ Reviewer)

ตารางนี้รวมทุกจุดใน pipeline ที่สามารถปรับแต่งเพื่อเพิ่มประสิทธิภาพ:

| ขั้นตอน | จุดปรับ | พารามิเตอร์ | ค่าปัจจุบัน | ผลกระทบ |
| --- | --- | --- | --- | --- |
| **Pose Estimation** | Video crop | ไม่มี (ต้องทำก่อน) | ไม่ได้ crop | Crop เฉพาะส่วนผู้แปล → ลด noise |
| **Segmentation** | Boundary threshold | `--sign-b-threshold` | 30 | ↑ = segment น้อยลง, ↓ = segment มากขึ้น |
| **Segmentation** | Onset threshold | `--sign-o-threshold` | 50 | ↑ = เลือกเฉพาะท่าชัด, ↓ = จับท่าเบาๆ ด้วย |
| **Pre-shift** | Start bias | `--pr_subs_delta_bias_start` | 1.3-2.6 | ปรับตาม median start offset ของผลลัพธ์ |
| **Pre-shift** | End bias | `--pr_subs_delta_bias_end` | 1.0-2.1 | ปรับตาม median end offset ของผลลัพธ์ |
| **Embedding** | SignCLIP model | `--sign_clip_model` | multilingual | BSL / multilingual / ASL |
| **Embedding** | Text source | `--pr_sub_path` | CC / Gloss | Gloss text ให้ผลดีกว่า CC text |
| **Embedding** | Tokenization | `--tokenize_text_embedding` | off | on = word-level similarity |
| **DP** | Duration penalty | `--dp_duration_penalty_weight` | 2 | ↑ = บังคับ duration match มากขึ้น |
| **DP** | Gap penalty | `--dp_gap_penalty_weight` | 8 | ↑ = หลีกเลี่ยง group ที่มี gap |
| **DP** | Similarity weight | `--similarity_weight` | 6 | ↑ = เชื่อ embedding มากขึ้น |
| **DP** | Window size | `--dp_window_size` | 40 | ↑ = search space กว้างขึ้น |
| **DP** | Max gap | `--dp_max_gap` | 6 | จำกัดจำนวน segment ต่อ group |
| **Post-process** | Overlap fix | `fix_overlap_vtt.py` | clamp end | ลด overlap จาก 88% → 0% |

> **วิธีใช้ตาราง:** ดูจาก evaluation ว่าปัญหาอยู่ตรงไหน → ปรับจุดที่เกี่ยวข้อง → รัน → วัดผลซ้ำ

## 7. ขั้นตอนที่ 3 — Segmentation

**สถานะ: ✅ เสร็จสมบูรณ์**

### สิ่งที่ทำ

ใช้ **linguistic segmenter** (โมเดล E4s-1) ตรวจจับตำแหน่งท่ามือแต่ละท่าจากข้อมูล pose โดยใช้ threshold สำหรับแยกว่าช่วงไหนเป็น "กำลังแสดงท่ามือ" (SIGN) และช่วงไหนเป็น "หยุดพัก"

### หลักการทำงาน (รายละเอียดสำหรับ Reviewer)

Segmenter ใช้โมเดล **E4s-1** (EMNLP 2023 Findings) ที่ train บนข้อมูล BSL (BOBSL dataset):

1. **Input:** ไฟล์ `.pose` (skeleton 543 landmarks × ทุกเฟรม)
2. **โมเดล E4s-1:** binary classifier ที่ทำนายว่าแต่ละเฟรมเป็น "กำลังแสดงท่ามือ" หรือ "หยุดพัก/เปลี่ยนท่า"
3. **Boundary detection:** ใช้ 2 threshold ตัดสินใจ:
   - `sign-b-threshold` (boundary): เฟรมที่ probability > threshold นี้ = จุดเปลี่ยนท่ามือ
   - `sign-o-threshold` (onset): เฟรมที่ probability > threshold นี้ = จุดเริ่มท่ามือใหม่
4. **Output:** annotation 2 ระดับ — SIGN (ท่ามือแต่ละท่า) และ SENTENCE (กลุ่มท่ามือที่เป็นประโยค)

> **🔧 จุดที่ปรับได้:**
> - `--sign-b-threshold`: ค่าสูง → segment น้อยลง (merge ท่ามือเล็กเข้าด้วยกัน), ค่าต่ำ → segment ถี่ขึ้น
> - `--sign-o-threshold`: ค่าสูง → ตัด onset เฉพาะท่าที่ชัดเจน, ค่าต่ำ → จับท่ามือเบาๆ ด้วย
> - ⚠️ **สำคัญ:** threshold เหล่านี้ส่งผลต่อจำนวน sign segments ซึ่งส่งผลต่อ DP alignment โดยตรง
>   - segment มากเกินไป (over-segmentation) → DP มี candidate มาก, อาจจับคู่ผิด
>   - segment น้อยเกินไป (under-segmentation) → cue หลายตัวถูก assign ให้ segment เดียวกัน

## 8. ขั้นตอนที่ 4 — Alignment (ไม่ใช้ Embedding)

**สถานะ: ✅ เสร็จสมบูรณ์**

### สิ่งที่ทำ

รัน `align.py` แบบ `--similarity_measure none` — จับคู่ subtitle กับ sign segments โดยใช้ **Dynamic Programming (DP)** อิงจาก timing เพียงอย่างเดียว ไม่ใช้ semantic embedding

### หลักการทำงานของ DP Alignment (รายละเอียดสำหรับ Reviewer)

DP Alignment คือหัวใจของระบบ SEA ทำงานใน 5 ขั้นตอนย่อย:

#### 4.1 Pre-shift subtitles (Bias Correction)

ก่อน alignment ระบบจะ "เลื่อน" timestamp ของ subtitle ทั้งหมดไปข้างหน้า เนื่องจากคำบรรยายปกติ sync กับเสียงพูด แต่ท่ามือมาช้ากว่าเสมอ:

```
cue.start = cue.start + delta_bias_start   (default: +2.6s)
cue.end   = cue.end   + delta_bias_end     (default: +2.1s)
```

> **🔧 จุดที่ปรับได้ (Tunable):** `--pr_subs_delta_bias_start` / `--pr_subs_delta_bias_end`
> 
> **วิธีหา bias ที่เหมาะสม:** รัน experiment แรกด้วย bias=0 แล้วดู median start offset จากผลลัพธ์ นำค่านั้นมาเป็น bias ใน run ถัดไป (iterative refinement)

#### 4.2 Candidate Window Selection

สำหรับ cue แต่ละ cue ระบบจะ:
1. คำนวณ midpoint ของ cue → `cue_mid = (start + end) / 2`
2. คำนวณ midpoint ของ sign segments ทุกตัว
3. เลือก **W sign segments** ที่ midpoint ใกล้ cue_mid ที่สุด เป็น candidate

```
W = dp_window_size (default: 50)
```

> **🔧 จุดที่ปรับได้:** `--dp_window_size`
> 
> - W มากเกินไป → ช้า, อาจจับคู่ผิด (candidate ไกลจาก cue เกินไป)
> - W น้อยเกินไป → อาจพลาด sign segment ที่ถูกต้อง
> - **ค่าแนะนำ:** 30-50 สำหรับวิดีโอ 10-15 นาที

#### 4.3 Cost Matrix Computation

สำหรับทุกคู่ (cue_i, sign_group[k:j]) คำนวณ cost:

```
cost(cue_i, group[k:j]) = |cue_start − group_start|     ← start offset
                         + |cue_end − group_end|         ← end offset
                         + W_dur × |cue_dur − group_dur|  ← duration mismatch
                         + W_gap × total_gap_in_group     ← gap ระหว่าง sign segments ในกลุ่ม
                         + W_sim × (−similarity_total)     ← ยิ่งคล้ายกัน ยิ่ง cost ต่ำ
```

พารามิเตอร์ที่ควบคุมน้ำหนักของแต่ละ term:

| Parameter | CLI Flag | Default | ผลกระทบ |
| --- | --- | --- | --- |
| W_dur | `--dp_duration_penalty_weight` | 1.0 | สูง → บังคับให้ cue กับ group มี duration ใกล้กัน |
| W_gap | `--dp_gap_penalty_weight` | 5.0 | สูง → ลงโทษ group ที่มีช่องว่างระหว่าง segments |
| W_sim | `--similarity_weight` | 10.0 | สูง → ให้ความสำคัญกับ semantic similarity มากขึ้น |
| max_gap | `--dp_max_gap` | 10.0 | จำกัดจำนวน sign segments สูงสุดต่อ group |

> **🔧 จุดที่ปรับได้:** ทั้ง 4 พารามิเตอร์ข้างบน
>
> **กลยุทธ์ปรับ:**
> - ถ้า alignment กระจายมากเกินไป → เพิ่ม `W_dur` (บังคับ duration match)
> - ถ้ามี overlap สูง → เพิ่ม `W_gap`, ลด `max_gap`
> - ถ้า semantic matching ไม่ช่วย → ลด `W_sim` (หรือใช้ `--similarity_measure none`)
> - ถ้า embedding ดี → เพิ่ม `W_sim` เพื่อให้ DP เชื่อ embedding มากขึ้น

#### 4.4 DP Optimization (Numba JIT)

ใช้ dynamic programming 2D (M×N matrix) หาเส้นทาง assignment ที่มี total cost ต่ำสุด:

```
dp[i][j] = min over k ∈ [i-1, j) of {
    dp[i-1][k] + cost(cue_i, group[k:j])
}
```

โดย:
- `dp[i][j]` = cost รวมต่ำสุดที่ assign cue 1..i ให้ sign segment 1..j
- `prev[i][j]` = ค่า k ที่ให้ cost ต่ำสุด (ใช้สำหรับ backtrack)
- Inner loop ถูก compile ด้วย **Numba @njit** เพื่อความเร็ว

> **📊 Complexity:** O(M × W² × N) ใน worst case — ใช้เวลาไม่กี่วินาทีสำหรับ M=172, N=2780, W=50

#### 4.5 Backtracking & Output

1. หา `best_j = argmin(dp[M, :])` — sign segment สุดท้ายที่ถูก assign
2. Backtrack ผ่าน `prev[][]` เพื่อหา boundaries ของแต่ละ cue
3. **Post-processing:** สำหรับแต่ละ cue ลองหา sub-group ที่ดีที่สุดภายใน group ที่ถูก assign
4. อัปเดต `cue.start = group.start`, `cue.end = group.end`
5. เขียนผลเป็น VTT + EAF

- `segmentation_output/E4s-1_30_50/04_updated.eaf` — มี tier `SUBTITLE_SHIFTED`

## 10. ขั้นตอนที่ 6 — Alignment พร้อม Embedding

**สถานะ: ✅ เสร็จสมบูรณ์**

### หลักการ (รายละเอียดสำหรับ Reviewer)

เมื่อมี embedding แล้ว `align.py` จะคำนวณ **similarity matrix** ระหว่าง subtitle embedding กับ sign embedding แล้วรวมเข้ากับ DP cost function

#### กระบวนการคำนวณ Similarity Matrix

```
Input:
  subtitle_embedding : (M × 768) ← M cues, แต่ละ cue เป็น vector 768 มิติ
  sign_embedding     : (N × 768) ← N sign segments, แต่ละ segment เป็น vector 768 มิติ

Step 1: sim_matrix = dot(subtitle_embedding, sign_embedding.T)   → (M × N)
  sim_matrix[i][j] = ค่า similarity ระหว่าง cue_i กับ sign_j
  (ค่ายิ่งสูง = ยิ่งคล้ายกัน)

Step 2: Normalize ด้วย softmax (row-wise, default)
  → ทำให้แต่ละแถว (แต่ละ cue) มีค่ารวมเป็น 1

Step 3: Cumulative sum ตามแถว
  sim_cumsum[i][j] = sum of sim_matrix[i][0:j]
  → ใช้คำนวณ similarity_total ของ group[k:j] ได้เร็ว:
    similarity_total = sim_cumsum[i][j] - sim_cumsum[i][k]
```

#### ตัวเลือก Similarity Mode

| Mode | CLI Flag | ผลกระทบ |
| --- | --- | --- |
| **Pooled (default)** | ไม่ต้องระบุ | embed ทั้ง cue เป็น 1 vector → dot product กับ sign segment |
| **Tokenized (word-level)** | `--tokenize_text_embedding` | แยก cue เป็นรายคำ → embed แต่ละคำ → pool ด้วย max/mean |
| **None** | `--similarity_measure none` | ไม่ใช้ embedding (cost อิงจาก timing อย่างเดียว) |

> **🔧 จุดที่ปรับได้:**
> - เลือก mode: `none` (เร็ว, ไม่ต้องมี embedding), `sign_clip_embedding` (แม่นกว่า)
> - ปรับ `--similarity_weight`: สูง = เชื่อ embedding, ต่ำ = เชื่อ timing
> - เลือก text input: CC text (คำบรรยายภาษาไทย) vs Gloss text (คำอธิบายท่ามือ)
> - เลือก pooling: `--text_embedding_pooling max` (default) หรือ `mean`

### สูตร cost:

```
cost = |cue_start - group_start|
     + |cue_end - group_end|
     + duration_penalty × |cue_duration - group_duration|
     + gap_penalty × gap
     + similarity_weight × (-similarity_total)
```

## 13. ผลการเปรียบเทียบทุก Experiment

### ตารางผลรวม (เทียบกับ CC_Aligned ground truth — 119 entries)

| Experiment | Match | Mean Offset | Median Offset | Stdev | ±1s | ±2s | ±3s | Overlap |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| **B2** (BSL, CC text, tuned) | 69/172 | +1.02s | -0.05s | 5.53s | 74% | 96% | 97% | 88.3% |
| **B_MULTI** (Multilingual, CC text) | 69/172 | +0.91s | -0.05s | 5.50s | 78% | **97%** | **99%** | 88.3% |
| **C_MULTI** (Multilingual, Gloss text) | 69/172 | **+0.49s** | **-0.15s** | 5.54s | **80%** | 96% | **99%** | **87.7%** |
| C_MULTI_word (Multilingual, Gloss, word-level) | 69/172 | +0.51s | -0.15s | 5.48s | 77% | 96% | **99%** | 88.3% |
| D_ASL (ASL, CC text) | 69/172 | +1.25s | +0.37s | 5.54s | 59% | 81% | 96% | 88.9% |
| D_ASL_gloss (ASL, Gloss text) | 69/172 | +0.77s | -0.10s | 5.50s | 64% | 91% | 97% | 88.3% |
| D_ASL_word (ASL, Gloss, word-level) | 69/172 | +0.78s | -0.11s | 5.50s | 67% | 93% | 96% | 87.7% |

### การวิเคราะห์

1. **Multilingual model ดีกว่า BSL model** สำหรับภาษามือไทย:
   - B_MULTI ดีกว่า B2: ±2s ดีขึ้น 1%, ±3s ดีขึ้น 2%
   - Mean offset ลดลง 0.11s

2. **C_MULTI (Multilingual + Gloss text) ให้ผลดีที่สุด**:
   - Mean offset ลดจาก +1.02s → +0.49s (ลดลง ~52%)
   - 80% ของ cues อยู่ภายใน ±1 วินาที
   - 99% ของ cues อยู่ภายใน ±3 วินาที

3. **Word-level similarity ไม่ช่วยอย่างชัดเจน** ในกรณีนี้:
   - C_MULTI vs C_MULTI_word: ผลใกล้เคียงกัน

4. **ASL model ด้อยกว่า Multilingual** สำหรับ TSL:
   - ASL trained เฉพาะ ASL data จึง generalize มา TSL ได้ไม่ดี
   - D_ASL mean offset +1.25s (เทียบ B_MULTI +0.91s), ±1s เพียง 59% (เทียบ 78%)

5. **Overlap ยังสูง (~88%)** ในทุก experiment:
   - เกิดจาก 172 CC cues แต่ sign segments รองรับ ~119 slots
   - ต้องทำ post-processing เพื่อลด overlap

### ผลหลัง Post-processing (ลด Overlap)

| ไฟล์ | Overlap ก่อน | Overlap หลัง |
| --- | --- | --- |
| C_MULTI (`aligned_output_multi_gloss/04.vtt`) | 150/171 (87.7%) | **0/171 (0.0%)** |
| D_ASL_gloss (`aligned_output_asl_gloss/04.vtt`) | 151/171 (88.3%) | **0/171 (0.0%)** |

## 15. ปัญหาที่พบและวิธีแก้ไข

### 15.1 mediapipe version ไม่ถูกต้อง

**ปัญหา:** mediapipe 0.10.22+ เปลี่ยน API ทำให้ `videos_to_poses` ทำงานไม่ได้
**แก้ไข:** ล็อก version เป็น `mediapipe==0.10.21`

### 15.2 UnicodeDecodeError: video_ids.txt

**ปัญหา:** คำสั่ง `echo 04 > video_ids.txt` บน Windows สร้างไฟล์ UTF-16 BOM ซึ่ง Python อ่านไม่ได้
**แก้ไข:** สร้างไฟล์ด้วย Python:
```python
open('video_ids.txt', 'w', encoding='utf-8').write('04\n')
```

### 15.3 FileNotFoundError: path มี single quote

**ปัญหา:** `shlex.quote()` ใส่ single quote รอบ path ซึ่ง Windows cmd.exe ไม่รู้จัก
**แก้ไข:** เปลี่ยน `segmentation.py` เป็น `subprocess.run(shlex.split(cmd), shell=False)`

### 15.4 unrecognized argument: --subtitle_dir

**ปัญหา:** argument ที่ถูกต้องคือ `--pr_sub_path`
**แก้ไข:** ใช้ `--pr_sub_path` แทน

### 15.5 ModuleNotFoundError: numba, beartype

**ปัญหา:** dependency ไม่ได้ติดตั้ง
**แก้ไข:** `pip install numba beartype tqdm scikit-learn tabulate`

### 15.6 PyTorch ไม่ detect GPU (RTX 5060 Ti Blackwell)

**ปัญหา:** RTX 5060 Ti เป็น Blackwell architecture (sm_120) ต้องการ CUDA 12.8+
PyTorch cu126 จะ fallback ไป CPU โดยอัตโนมัติ
**แก้ไข:** ติดตั้ง PyTorch cu128:
```powershell
pip install --force-reinstall torch --index-url https://download.pytorch.org/whl/cu128
```

### 15.7 Overlap สูง ~88% ในผล alignment

**ปัญหา:** 172 CC cues แต่ sign segments รองรับ ~119 slots → cue ซ้อนกัน
**แก้ไข:** สร้าง `fix_overlap_vtt.py` ที่ clamp end time → overlap 0%
